# Requirements Document

## Introduction

This document specifies requirements for adding party-level chronicle sheet filling functionality to a Foundry VTT plugin for Pathfinder Society. The feature enables GMs to fill out chronicle sheet information once for all party members through the Party sheet interface, reducing repetitive data entry while maintaining support for player-specific information.

## Glossary

- **Chronicle_Sheet**: A PDF document that records a player's participation in a Pathfinder Society scenario, including rewards earned
- **Party_Sheet**: The Foundry VTT interface that displays all player characters in the current game session
- **GM**: Game Master, the user who runs the game and fills out chronicle sheets
- **Player_Character**: A character controlled by a player in the game
- **Layout_File**: A configuration file generated by chronicle2layout that specifies field positions on chronicle sheet PDFs
- **Season**: A grouping of chronicle layouts, typically corresponding to a Pathfinder Society season (e.g., "Season 5", "Season 6"). Seasons are retrieved from LayoutStore.getSeasons() and are used to filter the available layouts
- **Layout**: A specific chronicle sheet template within a season. Layouts are retrieved from LayoutStore.getLayoutsByParent(seasonId) based on the selected season
- **Shared_Field**: A chronicle sheet field with the same value for all party members (e.g., XP, event code, season, layout)
- **Unique_Field**: A chronicle sheet field with different values per player (e.g., treasure earned, items purchased)
- **Chronicle_Data**: The collection of information to be written onto chronicle sheets

## Requirements

### Requirement 1: Party-Level Chronicle Interface

**User Story:** As a GM, I want to access chronicle sheet filling from the Party sheet, so that I can fill out information for all players at once.

#### Acceptance Criteria

1. WHEN the GM opens the Party_Sheet, THE Party_Sheet SHALL display a "Society" tab to initiate party-level chronicle filling
2. WHEN the GM activates the Society tab, THE Plugin SHALL display a party chronicle filling interface
3. THE Party_Chronicle_Interface SHALL list all Player_Characters currently in the party
4. THE Party_Chronicle_Interface SHALL display fields for Chronicle_Data entry

### Requirement 2: Shared Field Management

**User Story:** As a GM, I want to enter shared information once, so that it applies to all party members automatically.

#### Acceptance Criteria

1. THE Party_Chronicle_Interface SHALL provide input fields for Shared_Fields
2. WHEN the GM enters a value in a Shared_Field, THE Plugin SHALL apply that value to all Player_Characters in the party
3. THE Plugin SHALL identify Shared_Fields including scenario name, event code, event date, XP earned, reputation earned, downtime earned, season selection, and layout selection
4. THE Party_Chronicle_Interface SHALL visually distinguish Shared_Fields from Unique_Fields
5. WHEN the GM selects a Season, THE Plugin SHALL filter the available Layout options to show only layouts belonging to that season (using LayoutStore.getLayoutsByParent(seasonId))

### Requirement 3: Unique Field Management

**User Story:** As a GM, I want to enter player-specific information separately, so that each player receives their individual rewards.

#### Acceptance Criteria

1. THE Party_Chronicle_Interface SHALL provide input fields for Unique_Fields for each Player_Character
2. WHEN the GM enters a value in a Unique_Field for a Player_Character, THE Plugin SHALL apply that value only to that Player_Character
3. THE Plugin SHALL identify Unique_Fields including treasure earned, items purchased, and boons received
4. THE Party_Chronicle_Interface SHALL display Unique_Fields grouped by Player_Character

### Requirement 4: Chronicle Generation from Party Data

**User Story:** As a GM, I want to generate all chronicle sheets from the party interface, so that I don't need to visit individual character sheets.

#### Acceptance Criteria

1. WHEN the GM completes Chronicle_Data entry in the Party_Chronicle_Interface, THE Plugin SHALL provide a control to generate chronicles
2. WHEN the GM triggers chronicle generation, THE Plugin SHALL create a Chronicle_Sheet for each Player_Character using the entered Chronicle_Data
3. FOR EACH Player_Character, THE Plugin SHALL combine Shared_Fields and that character's Unique_Fields into their Chronicle_Sheet
4. WHEN chronicle generation completes, THE Plugin SHALL notify the GM of success or failure for each Player_Character

### Requirement 5: Layout File Integration

**User Story:** As a GM, I want the party chronicle feature to use existing layout files, so that chronicle sheets are formatted correctly.

#### Acceptance Criteria

1. WHEN generating chronicles from party data, THE Plugin SHALL use Layout_Files to position Chronicle_Data on Chronicle_Sheets
2. THE Plugin SHALL apply the same Layout_File processing for party-generated chronicles as for individual character chronicles
3. IF a Layout_File is missing for a Chronicle_Sheet, THEN THE Plugin SHALL display an error message to the GM

### Requirement 6: Data Validation

**User Story:** As a GM, I want the system to validate my entries, so that I catch errors before generating chronicles.

#### Acceptance Criteria

1. WHEN the GM enters Chronicle_Data, THE Plugin SHALL validate that required Shared_Fields are populated
2. WHEN the GM enters Chronicle_Data, THE Plugin SHALL validate that required Unique_Fields are populated for each Player_Character
3. IF validation fails, THEN THE Plugin SHALL display error messages indicating which fields need correction
4. THE Plugin SHALL prevent chronicle generation while validation errors exist

### Requirement 7: Player Access to Chronicles

**User Story:** As a player, I want to download my chronicle from my character sheet, so that I can maintain my records.

#### Acceptance Criteria

1. WHEN a Chronicle_Sheet is generated from the Party_Chronicle_Interface, THE Plugin SHALL attach it to the corresponding Player_Character
2. WHEN a player opens their character sheet, THE Plugin SHALL display available chronicles for download
3. THE Plugin SHALL maintain the same chronicle download functionality for party-generated chronicles as for individually-generated chronicles

### Requirement 8: Data Persistence

**User Story:** As a GM, I want my partially-filled chronicle data to be saved, so that I can complete it later if interrupted.

#### Acceptance Criteria

1. WHILE the Party_Chronicle_Interface is open, THE Plugin SHALL save Chronicle_Data as the GM enters it
2. WHEN the GM closes and reopens the Party_Chronicle_Interface, THE Plugin SHALL restore previously entered Chronicle_Data
3. WHEN chronicles are successfully generated, THE Plugin SHALL NOT clear the saved Chronicle_Data (data persists for review and correction)
4. WHEN the GM clicks the Clear button, THE Plugin SHALL clear all saved Chronicle_Data
5. WHEN the GM clicks the Clear button, THE Plugin SHALL reset the event date field to today's date
